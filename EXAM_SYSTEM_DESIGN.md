# 销售在线考试系统改造设计文档

## 目标

把当前“发布考试时顺手维护题目”的模式，改造成更适合长期运营的两层结构：

1. **基础题库先维护**：题目作为独立资产存在，支持单选、多选、标签、类目、难度、解析、单独新增、删除、导入、导出。
2. **发布考试再组卷**：发布考试时从基础题库中筛选并勾选试题组成试卷，不再自动生成默认题，避免题目资产重复和维护混乱。
3. **保持当前界面风格**：沿用现有精致紧凑的考试页面布局，在当前“预览 + 右侧考试列表 + 报表”的基础上增强，而不是推倒重做。

## 信息架构

- **考试首页**
  - 顶部 KPI：进行中考试、题库总量、平均通过率、需补考人数。
  - 左侧主区：当前选中考试预览，展示真实试题。
  - 右侧：考试列表，可发布、参加考试、查看题库。
  - 下方：成绩统计、补考提醒、题型结构。

- **基础题库维护**
  - 入口：考试页顶部“题库维护”或考试列表中的“题库”。
  - 支持动作：新增题、导入 Excel/CSV、导出 Excel、删除题。
  - 题目字段：题干、A-F 选项、正确答案、题型、类目、难度、标签、解析。
  - 列表体验：按类目/标签/题型筛选，题目卡片可直接删除。

- **发布考试**
  - 填写考试名称、类目、考试时长、及格分、适用对象。
  - 从基础题库勾选题目，可按类目/标签过滤。
  - 右侧实时显示已选数量、单选/多选结构和难度结构。
  - 确认后生成考试，考试与题库题目建立引用关系。

## 数据设计

- `exam_questions`
  - 继续作为基础题库表。
  - `exam_id` 允许为空或使用 `bank` 表示基础题库归属。
  - 新增 `tags_json` 存储标签数组。
  - 原历史题目保留，兼容旧考试。

- `exam_question_links`
  - 新增考试与题目关联表。
  - 字段：`exam_id`, `question_id`, `sort_order`。
  - 发布考试时写入关联关系。

## 接口设计

- `GET /api/exam-questions`
  - 获取基础题库，可带 `category`、`tag`、`type`。

- `POST /api/exam-questions`
  - 新增基础题。

- `DELETE /api/exam-questions/:id`
  - 删除基础题，并清理组卷引用。

- `POST /api/exam-questions/import`
  - 导入基础题库。

- `GET /api/exam-questions/export`
  - 导出基础题库 JSON 数据，前端转 Excel。

- `POST /api/exams`
  - 发布考试时接收 `questionIds`，按勾选题目组卷。

## 三轮自我审核

### 第一轮：产品体验审核

- 问题：如果题库维护还藏在某个考试下面，用户会误以为题目只属于这个考试。
- 修正：题库维护必须变成“全局基础题库”，考试只是引用题目。
- 结论：顶部“题库维护”进入全局题库，考试列表中的“题库”也打开同一个题库，但默认带当前考试类目筛选。

### 第二轮：数据一致性审核

- 问题：如果考试复制题目，后续基础题更新无法同步；如果考试直接读实时题库，历史考试可能被修改影响成绩追溯。
- 修正：当前阶段采用“考试引用题目”的轻量方案，保留题目 ID 与当前题目内容。后续如需强审计，可增加发布快照。
- 结论：先满足可用性和维护效率，删除题目时清理引用；已发布考试若题目被删，预览和答题自动忽略失效题。

### 第三轮：运营完整性审核

- 问题：只支持新增题和发布考试不够，实际运营需要批量导入、导出备份、标签筛选、多选题、删除错题。
- 修正：本次必须一起实现导入、导出、标签、删除、多选、勾选组卷。
- 结论：以“基础题库资产化”为定稿；报表继续按真实题库和考试成绩计算。

## 最终定稿

本次开发按以下顺序落地：

1. 后端补基础题库字段、接口和组卷关联。
2. MySQL schema 与兼容迁移同步更新。
3. 前端新增题库维护大弹窗，支持新增、多选、标签、删除、导入、导出。
4. 前端发布考试弹窗改为勾选题目组卷。
5. 补后端自检和 Playwright 页面级测试。
